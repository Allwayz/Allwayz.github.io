Java基础-0x0c:线程池底层原理

发表于 2020-06-02
字数统计:

JUC

线程池

线程池底层工作原理

  1. 在创建了线程池后，等待调过来的任务请求。

  2. 当调用execute()方法添加一个请求任务时，线程池会做如下判断：

    1. 如果正在运行的线程数量小于corePoolSize，那么马上创建线程运行这个任务；
    2. 如果正在运行的线程数大于或等于corePoolSize，那么将这个任务放入队列；
    3. 如果这个时候队列满了且正在运行的线程数量还小于maximumPoolSize,那么还是要创建非核心线程立刻来运行这个任务；
    4. 如果队列满了且正在运行的线程数量大于或等于maximumPoolSize，那么线程池会启动饱和拒绝侧裂来执行。

  3. 当一个线程完成任务时，它会从队列中取下一个任务来执行。

  4. 当一个线程无事可做超过一定时间(keepAliveTime)时，线程池会判断：

    如果当权运行的线程数大雨corePoolSize，那么这个线程就会被停掉。

    所以线程池的所有任务完成后他最终会收缩到corePoolSize的大小。

线程池拒绝策略

AbortPolicy (Default)

直接抛出异常(RejectedExecutionException)，组织系统正常运行。

CallerRunsPolicy

“调用者运行”一种调节机制，该策略即不会抛弃任务，也不会抛出异常，而是将某些人物回退到调用方

DiscardOldestPolicy

抛弃队列中等待最久的任务，然后把当前任务加入队列中尝试再次提交当前任务

DescardPolicy

直接丢弃任务，不与任何处理也不抛异常。如果允许任务丢失，这是最好的一种方案。

以上策略均实现了RejectExecutionHandle接口

手写线程池

线程池参数配置

设置maximumPoolSize的值：

  • CPU密集型

    任务需要大量的运算，且没有阻塞，CPU一直全速运行

    CPU密集型只有在真正的多核CPU上才可能得到加速（通过多线程）

    一般公式： ==CPU核数 = 线程池内线程数==

  • I/O密集型

    • 由于I/O密集型任务线程并不是一直在执行任务，则应配置尽可能多的线程，如==CPU核数*2==

    • 该任务需要大量的I/O，即大量的阻塞

      在单线程上运行I/O密集型的任务会导致大量的浪费CPU的运算能力浪费在等待。所以在I/O密集型任务重使用多线程可以大大的加速程序运行，即使在单核CPU上，这种加速主要就是利用了被浪费掉的阻塞时间

      参考公式：==CPU核数/（1 - 阻塞系数）== （阻塞系数在0.8～0.9之间）

死锁

两个或两个以上的进程在执行过程中，因争夺资源而造成的一种互相等待的现象，若无外力干涉，那他们都将无法推进下去，如果系统资源冲粗，进程的资源请求都能得到满足，死锁出现的可能性就很低，否则就会因争夺有限的资源而陷入死锁

Demo：

主要原因

  • 系统资源不足
  • 进程运行推进的
  • 资源分配不当

排查方法

使用命令行排查：

  1. jps -l

    查看当前运行的所有进程，定位进程编号

  2. jstack {进程编号}

    查看某进程的所有异常 ，找到死锁查看
